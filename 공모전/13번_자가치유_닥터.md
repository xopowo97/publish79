## 1. 에이전트 개요
*   **역할**: 12번 감시_눈치왕이 Gemini API로 분석·생성한 에러 분석 JSON 보고서를 수신하여, GitHub API를 통해 `script.js` 소스코드를 격리 분석하고, 자가치유 패치 코드를 자율 생성한 뒤 GitHub Pull Request(PR)를 자동 발행한다.
*   **목적**: 에러 발생 → 자동 분석 → 자동 패치 PR 발행까지의 전 과정을 대표님의 수동 코딩 개입 없이 완전 자율화하여, 서비스 중단 시간(MTTR)을 0에 수렴시킨다.
*   **핵심 철학**: 이 에이전트는 Gemini 웹 상의 '수동 Gem'이 아니라, Vercel 서버리스 환경에서 Google Gemini API REST 및 GitHub REST API와 직접 다이렉트 통신하는 **완전 자율형 API 파이프라인** 구조로 설계된다.

## 3. 세부 구현 로드맵

### ✅ Step 1: 9번 에러 분석 JSON 수신 및 파싱
*   12번 감시_눈치왕(`api/send-error.js`)이 Gemini API로 생성한 에러 분석 보고서 JSON(`{ errorType, rootCause, affectedLine, healStrategy }`)을 `api/self-heal.js`의 POST 요청 바디로 수신한다.
*   수신 즉시 필수 필드 존재 여부를 검증하고, 누락 시 `400 Bad Request` 응답으로 조기 탈출하여 파이프라인 오염을 차단한다.

### ✅ Step 2: GitHub API로 소스코드 격리 조회
*   GitHub REST API `GET /contents/script.js`를 호출하여 현재 프로덕션 브랜치의 `script.js` 원문을 Base64 디코딩하여 취득한다.
*   조회 실패 시(404/401) 즉시 에러 상태를 `deploy_status` 테이블에 기록하고, 디스코드 알림으로 대표님께 보고한 뒤 안전 종료한다.

### ✅ Step 3: Gemini API 자가치유 패치 코드 자율 생성
*   취득한 `script.js` 원문과 9번 에러 분석 JSON을 Gemini API의 `contents[].parts[].text`로 조합하여 13번 자가치유_닥터 페르소나(`systemInstruction`)와 함께 투입한다.
*   **System Instruction 핵심**: *"너는 출판친구 플랫폼의 코드수정_닥터이다. 제공된 에러 분석 보고서와 소스코드를 기반으로, 오직 최소한의 수정만으로 버그를 제거하는 안전한 패치 코드 diff를 JSON 형식으로만 출력하라. 기존 비즈니스 로직, 주문 모듈, UI 디자인은 1바이트도 건드리지 마라."*
*   Gemini가 출력한 패치 코드 JSON(`{ patchedContent, commitMessage, branchName }`)을 파싱한다.

### ✅ Step 4: GitHub PR 자동 발행 및 Supabase 상태 기록
*   Gemini가 생성한 `branchName`(`heal/auto-fix-{timestamp}`)으로 GitHub에 신규 브랜치를 자동 생성한다.
*   패치된 `script.js` 내용을 Base64 인코딩하여 `PUT /contents/script.js`로 자동 커밋한다.
*   `POST /pulls`로 PR을 자동 발행하고, 생성된 PR URL을 Supabase `deploy_status` 테이블(`pr` 컬럼)에 기록한다.
*   11번 배포_배달이에게 승인 요청 URL을 디스코드로 자동 전송하여 대표님의 원클릭 모바일 결재를 대기한다.

## 5. API 기반 자율형 파이프라인 연동 설계 (Autonomous API Pipeline Design)

*   **연동 주체**: Vercel 서버리스 환경의 `api/self-heal.js` — 외부 npm 라이브러리 없이 Node.js 표준 `fetch`로 Google Gemini API 및 GitHub REST API와 직접 REST 통신.
*   **파이프라인 트리거**: 12번 감시_눈치왕(`api/send-error.js`)이 Gemini 분석 완료 후 `api/self-heal.js`로 POST 요청을 전송하는 시점.
*   **데이터 파이프라인 (입출력 덤프 흐름)**:

    ```
    [12번 감시_눈치왕]
        ↓ POST /api/self-heal
        ↓ Body: { errorType, rootCause, affectedLine, healStrategy, userId }
        ↓
    [13번 자가치유_닥터 — api/self-heal.js]
        │
        ├─ Step 1: 수신 JSON 파싱 및 필드 검증
        │
        ├─ Step 2: GitHub GET /contents/script.js
        │              → Base64 디코딩 → scriptSource (원문 취득)
        │
        ├─ Step 3: Gemini API POST (10번 페르소나 + scriptSource + 에러 분석 JSON)
        │              → 응답: { patchedContent, commitMessage, branchName }
        │
        ├─ Step 4-A: GitHub POST /git/refs (신규 치유 브랜치 생성)
        ├─ Step 4-B: GitHub PUT /contents/script.js (패치 코드 자동 커밋)
        ├─ Step 4-C: GitHub POST /pulls (PR 자동 발행)
        │              → prUrl 취득
        │
        ├─ Step 4-D: Supabase POST /rest/v1/deploy_status
        │              → { pr: prUrl, status: 'PENDING_REVIEW' }
        │              → Prefer: return=minimal (v3 방어 패치 기준)
        │
        └─ Step 4-E: 디스코드 웹훅 → 11번 배포_배달이에게 PR 승인 요청 전달
                         → [🔧 13번 자가치유_닥터 자가치유 PR 발행 완료] Embed 전송
                         → 대표님 원클릭 모바일 결재 URL 포함
        ↓
    [11번 배포_배달이]
        → 대표님 디스코드 승인 터치 → /api/deploy-approval 호출 → 최종 배포 실행
    ```

*   **12번 보안통제_보안관 연계 지점**: 10번이 생성한 패치 코드(`patchedContent`)는 GitHub PR 형태로 격리되며, 12번 보안통제_보안관이 PR 코드를 스캔하여 `SAFE` 판정을 내리기 전까지 프로덕션 브랜치에 머지되지 않는다. 거버넌스 락(Governance Lock) 구조 준수.
*   **주의 사항**: 이 문서는 13번 자가치유_닥터 에이전트를 Gemini API로 빌딩할 때 삽입할 System Instruction 프롬프트의 기준서 역할을 하며, 어떠한 수동 코딩 프로세스도 포함하지 않는다.
