# 1번_딥서치_살피미.md

> **📌 GEM 01. 딥서치 살피미 (Deep Search Gatherer)**

## 1. 에이전트 개요
*   **역할**: 도서관 대출 통계 데이터(도서관 정보나루 등), 시중 온라인 서점 API(알라딘 등), 그리고 국립중앙도서관 국가서지 표준 데이터 및 한국저작권위원회 '공유마당' 오픈 API를 유기적으로 조회하여, [트랙 1: B2B 절판 도서] 및 [트랙 2: IP 저작권 만료 도서] 발굴에 최적화된 이원화(Two-Channel) 수집 클라이언트 모듈 및 프록시 서버리스 함수.
*   **목적**: 유저 또는 시스템이 요청한 카테고리에 맞춰 최적화된 데이터 수집 채널을 자율 선택(B2B는 인기 대출/재고 체크, IP는 공공 공유마당/1956년 이전 역사 서지 체크)하여 복간 후보군을 발굴하고, 15번 보안_보안관의 악성 키워드 인젝션 필터링 및 12번 감시_눈치왕의 에러 감지와 연계하여 데이터 무결성을 보장한다.
*   **핵심 철학**: "수집 목적에 부합하는 데이터 원천 다각화". 대출 인기 통계에만 의존해 70년 이상 경과한 고전 도서를 탐색하는 논리적 한계를 타파하기 위해, B2B용 시장성 검증 채널과 IP용 저작권 만료 데이터 수집 채널을 논리적으로 분리(이원화)하여 플랫폼 비즈니스 타당성을 수호한다.

## 2. 연동 상태 및 환경 변수 정보

*   **국립중앙도서관 API 연동**: ✅ 완료 (`api/library.js` 내 Vercel 서버리스 프록시 및 kwd 파라미터 매핑 구현 완료)
    *   호출 엔드포인트: `https://www.nl.go.kr/NL/search/openApi/search.do`
    *   npm 패키지 없이 Node.js 표준 `fetch` 다이렉트 REST 통신 방식을 고수하여 Vercel 환경 내의 종속성 충돌 예방.
*   **15번 보안_보안관(보안관) 1차 필터 연동**: ✅ 완료 (`api/library.js` 내 `runSecuritySheriff_Library` 가동)
    *   XSS (`<script>`, `javascript:`), SQL Injection (`union`, `select`), 프롬프트 인젝션 패턴 검출 즉시 API 호출 차단 및 패킷 파기.
*   **12번 감시_눈치왕 에러 보고 연동**: ✅ 완료 (API 통신 실패 및 DANGER 판정 시 `api/send-error` 서버-to-서버 즉각 호출 연동)
*   **Vercel 등록 환경 변수**:

    | 변수명 | 설명 | 상태 |
    |---|---|---|
    | `LIBRARY_API_KEY` | 국립중앙도서관 오픈 API 인증 키 | ✅ 등록 완료 |

## 3. 세부 구현 로드맵

### ✅ Step 1: 클라이언트 도서 검색 요청 수신
*   사용자 브라우저 또는 상위 에이전트가 `/api/library?keyword=...`로 도서 검색 요청을 전송한다.
*   로컬/실서버 환경 자동 분기(`LIBRARY_PROXY_ENDPOINT`)를 통해 호스트 이름에 따라 개발용 로컬 주소와 실서버 배포 주소로 분기 호출된다.

### ✅ Step 2: 12번 AI 보안관의 1차 클라이언트 입력 키워드 보안 스캔
*   API 서버리스 핸들러 진입 즉시 15번 보안_보안관 필터가 가동된다.
*   **DANGER 판정 시**: `400 Bad Request` 에러를 반환하고, 12번 감시_눈치왕(`api/send-error`)에게 위협 탐지 감사 로그를 포함한 에러 보고서를 서버-to-서버로 즉시 송출한 뒤 패킷을 파기한다.

### ✅ Step 3: 국립중앙도서관 오픈 API 실제 호출 및 응답 취득
*   검색어 파라미터명을 명세에 부합하는 `kwd`로 정상 매핑하여 국립중앙도서관 API 서버로 `fetch` 통신을 수행한다.
*   API 응답 실패(5xx, 4xx) 또는 네트워크 타임아웃 발생 시, 12번 감시_눈치왕에게 장애 내역을 즉각 자동 보고하고 클라이언트에 502/500 응답을 반환하여 파이프라인 데드락을 방지한다.

### ⏳ Step 4: 2번 다듬이 에이전트로 데이터 가공 전달 및 대시보드 연동
*   수신된 비정형 API 응답 구조에서 `libraryData.result`를 안전하게 1차 매핑하고, `libraryData.total` 값을 정수로 변환하여 전체 검색 건수를 수집한다.
*   데이터 타입 가드(`Array.isArray`)를 통해 데이터 구조 안전성을 확보한 상태로 2번 데이터 정제 에이전트 '다듬이'에 전달하여 표준 DB 규격 전처리를 수행하도록 연동한다.

## 5. API 기반 자율형 파이프라인 연동 설계 (Autonomous API Pipeline Design)

*   **연동 주체**: Vercel 서버리스 환경의 `api/library.js`, `api/pipeline.js` 및 클라이언트 `control.js` 내 `triggerCtrlPipeline`.
*   **파이프라인 트리거**: 클라이언트 브라우저 로드 시 UAT 검증 목적의 자동 호출, 또는 사용자가 대시보드 상에서 특정 카테고리를 선택하고 파이프라인을 실행하는 시점.
*   **데이터 파이프라인 (이원화 입출력 덤프 흐름)**:

    ```
    [사용자 카테고리/키워드 선택]
         │
         ├─ [일반/절판/B2B 성격의 분야] (소설, 역사, 경제경영 등)
         │    └─► [채널 1: B2B 시장 검증 수집] 
         │            도서관 정보나루 인기 대출 API (대출수 확보) ➔ 알라딘 API 교차 체크 (품절/절판 여부)
         │
         └─ [저작권 만료 / IP 성격의 분야] (저작권 만료 명저 등)
              └─► [채널 2: IP 공공 저작물 수집]
                      한국저작권위원회 공유마당 오픈 API ➔ 국립중앙도서관 서지정보 API (1956년 이전 출판 도서 필터링)
                            ※ 도서관 인기 대출 순위에 없는 잠겨있는 고전 명작을 직접 추적 및 발굴
         │
         ▼
    [1번 살피미 프록시 — api/library.js & api/pipeline.js]
        ├─ Step 1: 12번 AI 보안관 1차 키워드 스캔 (XSS, SQLi, Prompt Injection)
        │              → [DANGER 판정 시] ➔ 9번 에러 보고 ➔ 400 Bad Request 차단 (패킷 파기)
        │              → [SAFE 판정 시] ➔ 다음 단계 진행
        ├─ Step 2: 채널별 연동 API 실제 호출 (적용 자격증명: LIBRARY_API_KEY, GONGYU_MADANG_KEY 등)
        │              → [HTTP/네트워크 에러 발생 시] ➔ 12번 감시_눈치왕에게 자동 서버-to-서버 보고
        ├─ Step 3: API 성공 응답 수신 및 결과 파싱 (기본 메타데이터 및 카테고리 매핑)
        │              → 결과 데이터 보정 및 타입 가드 적용 (Array.isArray)
        └─ Step 4: 2번 다듬이 에이전트로 데이터 전송 및 클라이언트 UI 동적 시각화 연동
    ```

*   **2번 다듬이 에이전트 연계 지점**: 살피미가 안전하게 확보하여 정제한 검색 결과 리스트(`results`)와 전체 건수(`totalCount`)는 2번 '다듬이' 에이전트의 전처리 필터(ISBN 체크, 저작권 만료 여부 판별, 대출 횟수 통계 연동) 파이프라인에 입력값으로 투입된다.
*   **주의 사항**: 각 기관의 API 키 노출 방지를 위해 클라이언트 단에서 직접 외부 공공 데이터를 호출하는 코드는 절대 작성하지 않으며, 반드시 Vercel 서버리스 프록시 레이어를 경유하게 통제한다.
