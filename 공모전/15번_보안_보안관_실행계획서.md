# [12번 실시간 AI 보안관] 에러 필터링 및 보안 위협 차단 실행계획서

대표님의 지침에 따라 외부 위협으로부터 시스템을 보호하기 위해 **12번 실시간 AI 보안관(Security Sheriff)** 에이전트를 최우선으로 개발합니다. 

이 단계에서는 에러 수신 API(`/api/send-error.js`)에 도달하는 모든 데이터를 실시간 검사하여 인젝션(XSS, SQLi) 및 AI 우회 공격(Prompt Injection)을 차단하고 대표님께 즉각 경고를 보내는 방어막을 구축합니다.

---

## 1. 대표님 검토 요구 사항 (보안 제어)

> [!IMPORTANT]
> **1) 공격 탐지 시 즉시 차단 및 우회 알림**
> - 악성 페이로드(예: `<script>`, `UNION SELECT` 등)가 에러 메시지에 포함되어 들어올 경우, 구글 Gemini API 호출을 **즉시 차단**하여 API Key 사용량 낭비 및 프롬프트 오염을 방지합니다.
> - 대신, 디스코드 알림에는 **[🚨 보안 위협 감지: 12번 AI 보안관 긴급 보고]**라는 적색 임베드 경고로 대표님께 즉시 우회하여 전송합니다.
> 
> **2) 프롬프트 인젝션 방어 패턴**
> - 에러 메시지 내용에 Gemini API 시스템 지침을 초기화하거나 무력화하려는 악의적인 문구(예: "Ignore previous instructions", "이전 지침은 무시하고 다음을 수행하라")가 포함될 경우, 15번 보안_보안관 필터가 이를 '위험' 수준으로 탐지하고 예외 처리합니다.

#### [MODIFY] [supabase_setup.sql](file:///c:/Users/seo%20sang%20won/001.작업파일/004. 출판친구/018. 안티그래비티/scratch/supabase_setup.sql)
- **RLS (Row Level Security) 철통 잠금**: `agent_audit_logs` 및 `reprint_candidates` 테이블의 RLS를 활성화하고, anon(익명 사용자) 접근 정책을 `DROP POLICY`로 완전 폐기하여 외부 해킹 및 Anon Key 탈취 시도를 차단. 오직 비공개 `service_role` 비밀 키를 품은 Vercel API 백엔드만 접근하도록 통제.

- **안전한 스키마 마이그레이션**: `DO $$` 멱등성 블록을 구성하여 기존 DB 테이블의 `is_out_print` 컬럼 데이터를 신규 `is_out_of_print` 컬럼으로 자동 이전 후 구버전 컬럼 삭제. 누락된 `publisher`, `demand_index`, `updated_at` 컬럼 동적 추가 및 `pub_year` 타입 변경을 데이터 유실 없이 보정.

## 3. 검증 계획 (테스트 방법)

### 수동 테스트 및 시연 시나리오
1. **보안 공격 모의 테스트 (XSS & SQLi)**:
   - 개발자 도구 콘솔에서 강제로 악성 코드가 포함된 에러 함수를 실행합니다.
   - 디스코드 채널에 **"🚨 보안 위협 감지: 12번 AI 보안관 긴급 보고"**가 빨간색 Embed 테두리로 전송되고, 구글 Gemini API 호출이 완벽히 차단(로그로 확인)되는지 검증합니다.
2. **인프라 보안 RLS 차단 검증 (3단계 실서버 UAT)**:
   - 익명 사용자가 Supabase 클라이언트를 통해 `reprint_candidates` 테이블 직접 조회 시, RLS에 의해 조회되지 않고 데이터가 차단되는지 검증.
3. **서버리스 API Rate Limiting 검증**:
   - 백엔드 API에 분당 한도를 초과하는 비정상적인 반복 요청을 보낼 때, 즉시 `429 Too Many Requests` 상태와 함께 호출이 차단되는지 검증.
