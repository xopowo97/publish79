## 1. 에이전트 개요
*   **역할**: 대표님의 모바일 디스코드 결재 링크 터치 신호를 Vercel 서버리스 API(`api/deploy-approval.js`)로 수신하고, Supabase `deploy_status` 테이블에 승인/반려 상태를 즉시 기록한다. 최종 승인 확정 시 GitHub 머지 명령을 발행하여 실서버 반영을 총괄 지휘하는 자율 배포 사령탑이다.
*   **목적**: 13번 자가치유_닥터가 발행한 PR을 대표님이 모바일에서 단 한 번의 터치로 승인·반려할 수 있도록 결재 UI를 제공하고, 그 결과를 거버넌스 DB에 즉시 반영하여 전체 자율 배포 파이프라인의 최종 관문을 완전 자동화한다.
*   **핵심 철학**: 이 에이전트는 Gemini 웹 상의 '수동 Gem'이 아니라, Vercel 서버리스 환경에서 Supabase REST API 및 GitHub REST API와 직접 다이렉트 통신하는 **완전 자율형 API 파이프라인** 구조로 설계된다. 대표님의 터치 한 번이 코드 머지와 실서버 배포까지 연결되는 원클릭 거버넌스 구조가 이 에이전트의 존재 이유다.

## 3. 세부 구현 로드맵

### ✅ Step 1: 대표님 모바일 결재 링크 수신 및 파라미터 파싱
*   디스코드 Embed에 포함된 승인/반려 URL을 대표님이 모바일에서 터치하면, Vercel 서버리스 함수 `api/deploy-approval.js`가 GET 요청으로 호출된다.
*   URL 쿼리 파라미터 `action`(`approve` / `reject`)과 `pr`(PR URL 문자열)을 즉시 파싱한다.
*   파라미터 파싱 단계에서 어떤 예외가 발생해도 전역 최외곽 `try-catch`(외벽)가 낚아채어 `res.status(200).send()`로 안전 탈출 — 대표님 모바일 화면에 검은 화면이 절대 노출되지 않는다.

### ✅ Step 2: Supabase `deploy_status` 테이블 상태 기록
*   수신된 `action` 값에 따라 `targetStatus`를 `APPROVED` 또는 `REJECTED`로 결정한다.
*   Supabase `deploy_status` 테이블에 `{ pr, status }` 페이로드를 POST 기록한다.
*   **4중 방어 가드 체계** (v3 방어 패치 완결판):
    *   **가드 1**: 환경변수(`SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`) 존재 여부 검증
    *   **가드 2**: `startsWith('https://')` — 프로토콜 미포함 URL로 인한 `TypeError: Invalid URL` Fatal 원천 차단
    *   **가드 3**: `new URL(endpoint)` — URL 파싱 실제 통과 여부 사전 검증, 실패 시 fetch 실행 차단
    *   **가드 4**: `r.status` 코드만 확인, `r.text()` / `r.json()` body 이중읽기 절대 금지

### ✅ Step 3: 승인/반려 결과 HTML 응답 즉시 송출
*   Supabase 기록 완료 즉시 (약 0.1~0.5초 이내), 대표님의 모바일 화면에 결과 HTML 페이지를 렌더링한다.
*   **승인 시**: ✅ 초록 카드 — "🎉 출판친구 배포 승인이 완벽하게 성공했습니다!" 문구 출력.
*   **반려 시**: ❌ 빨간 카드 — "배포 반려 완료" 문구 출력.
*   HTML 내부 `.log` 패널에 Supabase DB 기록 결과(`HTTP 상태 코드` 포함)를 실시간 표시하여 투명한 처리 결과를 대표님께 즉시 보고한다.

### ✅ Step 4: 승인 확정 시 GitHub PR 자동 머지 및 실서버 반영 총괄
*   `action === 'approve'` 확정 시, GitHub REST API `PUT /pulls/{pull_number}/merge`를 호출하여 13번 자가치유_닥터가 발행한 패치 PR을 `squash` 방식으로 자동 머지한다.
*   머지 완료 즉시 Vercel의 GitHub 연동 훅이 자동으로 프로덕션 재배포를 트리거 — **완전 무인 실서버 반영 완료**.
*   머지 결과를 Supabase `deploy_status` 테이블에 `MERGED` 상태로 최종 업데이트하고, 디스코드 채널에 **[✅ 11번 배포_배달이 배포 완료 보고]** 최종 알림을 송출한다.

## 5. API 기반 자율형 파이프라인 연동 설계 (Autonomous API Pipeline Design)

*   **연동 주체**: Vercel 서버리스 환경의 `api/deploy-approval.js` — 외부 npm 라이브러리 없이 Node.js 표준 `fetch`로 Supabase REST API 및 GitHub REST API와 직접 REST 통신.
*   **파이프라인 트리거**: 대표님이 모바일 디스코드에서 승인/반려 URL을 터치하는 순간.
*   **데이터 파이프라인 (입출력 덤프 흐름)**:

    ```
    [대표님 모바일 디스코드]
        ↓ URL 터치: /api/deploy-approval?action=approve&pr={PR_URL}
        ↓
    [11번 배포_배달이 — api/deploy-approval.js]
        │
        ├─ 전역 외벽 handler() — 무조건 200 보장 (어떤 Fatal도 500 불가)
        │       ↓
        ├─ Step 1: URL 파라미터 action 및 pr 파싱
        │       ↓
        ├─ Step 2: Supabase POST /rest/v1/deploy_status (APPROVED 기록)
        │              → Prefer: return=minimal (v3 방어 패치 기준)
        │       ↓
        ├─ Step 3: 즉각적인 HTML 응답 (초록 카드 또는 빨간 카드 송출)
        │       ↓
        ├─ Step 4: GitHub PUT /pulls/{pull_number}/merge (squash merge)
        │       ↓
        ├─ Step 5: Vercel 배포 트리거 작동 (무중단 재배포 실행)
        │       ↓
        └─ Step 6: Supabase POST /rest/v1/deploy_status (MERGED 최종 기록)
                   → 디스코드 일반채널에 최종 배포 성공 보고
    ```

*   **주의 사항**: 이 문서는 14번 배포_배달이 에이전트를 Vercel 서버리스 API로 빌딩할 때 삽입할 System Instruction 프롬프트 및 설계 기준서 역할을 하며, 어떠한 수동 머지/배포 조작 프로세스도 포함하지 않는다.
